## 1. 评估方法与维度

本报告从以下八个维度对每种候选技术进行评分（★ = 1 分，☆ = 0.5 分，满分 5★）：

| 维度 | 权重 | 说明 |
|------|------|------|
| **性能与渲染能力** | 高 | 2D 渲染效率、绘制调用控制、动画系统性能 |
| **开发效率与团队适配** | 高 | 学习曲线、迭代速度、脚本热重载、调试工具 |
| **跨平台需求** | 中 | Steam/itch.io/Steam Deck/Web Demo 支持度 |
| **扩展性与维护性** | 高 | 模块化能力、代码组织、长期可维护性 |
| **社区与生态系统** | 中 | 文档、插件、第三方库、问题解决资源 |
| **成本与许可限制** | 中 | 引擎许可费、版税、订阅费、商业限制 |
| **关键特性匹配度** | 高 | 对 2.5D 方块系统、符箓绘制、五行战斗等特性的原生/易实现程度 |
| **工程成熟度** | 低 | 版本稳定性、生产环境验证程度 |

> 权重说明：基于单人/小型独立团队的开发场景，**开发效率** 和 **关键特性匹配度** 权重最高。

## 2. 游戏需求分析摘要

根据设计文档，提炼出对技术选型有决定性影响的核心需求：

### 2.1 渲染与图形需求

| 需求 | 规格 | 技术影响 |
|------|------|----------|
| 像素风格 | 320×180 内渲染，2×/3× 放大 | 需要整数缩放 / 像素完美渲染 |
| 2.5D 方块系统 | 45° 斜角俯视，三面纹理（上/左/右） | 自定义坐标变换、自定义 Sprite 排序、大量 Sprite 实例管理 |
| Y-Sort 排序 | 对角线方向 + 高度层双重排序 | 引擎内置 Y-Sort 支持度至关重要 |
| 8 层渲染体系 | 地面/结构/装饰/角色/特效/UI 等多层 | CanvasLayer 或类似分层机制 |
| 粒子/特效 | 符箓发光、五行特效、天气系统 | 粒子系统性能与灵活性 |
| 光照 | 每场景 ≤8 Light2D，带阴影 | 2D 光照系统支持度 |

### 2.2 逻辑与系统复杂度

| 需求 | 复杂度 | 技术影响 |
|------|--------|----------|
| 伤害公式 | 多层乘法（五行×防御×境界×暴击×增益） | 脚本执行效率需求中等 |
| 符箓绘制输入 | 实时手势/方向序列识别 | 输入系统灵活性、自定义判断逻辑 |
| 物品/装备系统 | 多层属性叠加（基础+词条+镶嵌+套装） | 数据驱动架构需求高 |
| 任务系统 | 多条件追踪、分支、失败条件 | 状态机/条件系统灵活性 |
| 灵田生长模拟 | 定时更新、环境参数影响 | 后台计时、数据刷新效率 |
| 篆路自动化 | 信号传输、逻辑门、条件触发 | 类似 Factorio 电路网络的实时计算 |

### 2.3 平台与交付需求

| 需求 | 目标 |
|------|------|
| 主平台 | Steam (PC) |
| 辅助平台 | itch.io, Steam Deck (Proton) |
| 可选平台 | Web Demo |
| 性能目标 | PC 60 FPS, 最低 30 FPS (集成显卡) |
| 本地化 | 中文 + 英文 |
| 发布形式 | 可执行安装包 |

## 3. 游戏引擎评估

### 3.1 候选引擎一览

| 引擎 | 版本 | 类型 | 许可 | 2D 定位 |
|------|------|------|------|---------|
| **Godot** | 4.x | 全功能 | MIT（开源免费） | 原生 2D 优先 |
| **Unity** | 2022 LTS / 6 | 全功能 | 按收入许可（Personal 免费） | 2D 工具完善 |
| **GameMaker** | 2024.x | 2D 专用 | 订阅制（Creator/Enterprise） | 2D 专用 |
| **MonoGame** | 3.8+ | 框架 | MIT（开源免费） | 底层 2D 框架 |

### 3.2 详评矩阵

#### Godot 4.x

| 维度 | 评分 | 说明 |
|------|------|------|
| 性能与渲染 | ★★★★☆ | 2D 渲染极轻量，TileMap 效率高；复杂场景下 Sprite 实例化管理需注意 |
| 开发效率 | ★★★★★ | GDScript 零编译迭代，热重载一流，信号系统与架构高度匹配 |
| 跨平台 | ★★★★★ | Windows/Linux/macOS/Web/Android/iOS，Steam Deck 兼容性良好 |
| 扩展维护 | ★★★★★ | 节点树架构天然模块化，SignalBus 模式与引擎信号系统无缝集成 |
| 社区生态 | ★★★★☆ | 社区增长迅速，但资源量仍不及 Unity；中文资源较有限 |
| 成本许可 | ★★★★★ | MIT 许可，零成本，无版税，无收入门槛 |
| 特性匹配 | ★★★★★ | Y-Sort 内置、CanvasItem 可见性控制、输入映射系统完美匹配需求 |
| 工程成熟度 | ★★★★☆ | Godot 4.x 已稳定（4.2+），但部分插件生态仍在迁移中 |

**关键匹配项：**
- 2.5D 方块系统 -> 自定义 Sprite 节点 + Y-Sort 可完美实现
- 信号系统 -> 与 SignalBus 架构天然一致
- 输入映射 -> InputMap 完全匹配
- 场景系统 -> 资源热加载/卸载与区域切换需求匹配
- **关键缺陷：** TileMap 原生不支持 2.5D 等距渲染，需自研渲染层

#### Unity 2D

| 维度 | 评分 | 说明 |
|------|------|------|
| 性能与渲染 | ★★★★★ | Sprite Atlas、SRP Batcher、GPU Instancing 性能极佳 |
| 开发效率 | ★★★★☆ | C# + 可视化编辑器强大，但编译等待影响迭代速度 |
| 跨平台 | ★★★★★ | 最成熟的跨平台支持，Steam 集成文档丰富 |
| 扩展维护 | ★★★★☆ | ECS/DOTS 可用于性能优化，但整体复杂度更高 |
| 社区生态 | ★★★★★ | 最大游戏引擎社区，资源/教程/插件最丰富 |
| 成本许可 | ★★★☆☆ | 个人免费但年收入超 20 万美金后需付费；许可政策近年动荡 |
| 特性匹配 | ★★★★☆ | Tilemap 支持 isometric，但 2.5D 方块仍需自研；无原生信号总线 |
| 工程成熟度 | ★★★★★ | 业界最成熟的 2D 游戏工程之一 |

**关键评估：** 对于本项目的 2D 像素风格需求，Unity 的 3D 渲染管线（URP/HDRP）完全用不上，属于"用牛刀杀鸡"。编辑器臃肿（>3GB），项目构建体积大，学习曲线陡峭。对于单人开发者，Unity 在单纯 2D 项目上的 ROI 低于 Godot。

#### GameMaker Studio 2

| 维度 | 评分 | 说明 |
|------|------|------|
| 性能与渲染 | ★★★★☆ | 专为 2D 优化，像素完美渲染原生支持 |
| 开发效率 | ★★★★☆ | GML 易学，拖拽+代码混合，迭代速度快 |
| 跨平台 | ★★★★☆ | 支持主流平台，但 Web 和 Steam Deck 支持弱于 Godot |
| 扩展维护 | ★★★☆☆ | GML 语言表达能力有限，复杂数据结构管理笨拙 |
| 社区生态 | ★★★★☆ | 2D 像素游戏社区庞大，教程丰富 |
| 成本许可 | ★★★☆☆ | Creator $4.99/月，Enterprise $79.99/月，有版税（免费版） |
| 特性匹配 | ★★★☆☆ | 无信号系统（需手动模拟），自定义坐标变换难以实现 |
| 工程成熟度 | ★★★★☆ | 成熟稳定，大量成功像素游戏案例 |

**关键评估：** GML 的类型系统和数据结构能力不足以支撑本项目的复杂 RPG 数据模型（多级属性、配方系统、任务条件）。无原生信号总线和事件总线机制，与架构设计不匹配。

#### MonoGame / FNA

| 维度 | 评分 | 说明 |
|------|------|------|
| 性能与渲染 | ★★★★★ | 底层控制，完全优化空间，极致性能 |
| 开发效率 | ★★☆☆☆ | 无编辑器，无场景系统，一切代码构建 |
| 跨平台 | ★★★★★ | 原生 C#，广泛平台支持（FNA 尤其适合 Linux） |
| 扩展维护 | ★★★☆☆ | 代码级控制，良好维护性，但重构成本高 |
| 社区生态 | ★★★☆☆ | 小众，资源有限（Stardew Valley / Celeste 使用） |
| 成本许可 | ★★★★★ | MIT 许可，完全免费 |
| 特性匹配 | ★★☆☆☆ | 需自建场景系统、编辑器、资源管线，工作量巨大 |
| 工程成熟度 | ★★★★☆ | 稳定成熟，但工具链需要自行搭建 |

**关键评估：** 虽然 Stardew Valley 和 Celeste 使用 MonoGame/FNA，但它们的团队有经验和资源自建工具链。对于单人开发者，从零搭建相当于在游戏开发之前先做一个"引擎开发"项目，风险极高。

## 4. 编程语言评估

### 4.1 候选语言一览

| 语言 | 适用引擎 | 类型系统 | 性能层级 | 学习曲线 |
|------|----------|----------|----------|----------|
| **GDScript** | Godot | 动态（可选静态） | 中等 | 低 |
| **C#** | Godot / Unity / MonoGame | 强类型静态 | 高 | 中 |
| **C++ (GDExtension)** | Godot | 强类型静态 | 最高 | 高 |
| **GML** | GameMaker | 弱类型动态 | 中等 | 低 |
| **Visual Script** | Godot / Unity | 可视化 | N/A | 低（但维护性差） |

### 4.2 详评

#### GDScript

| 维度 | 评分 | 说明 |
|------|------|------|
| 性能 | ★★★☆☆ | 解释执行，数值密集运算（如伤害公式批量计算）可能成为瓶颈 |
| 开发效率 | ★★★★★ | Python 风格语法，零编译，热重载，引擎 API 零摩擦 |
| 维护性 | ★★★★☆ | 可选类型注解提升可维护性，函数 ≤40 行规范可保代码质量 |
| 生态 | ★★★★☆ | Godot 原生支持，但独立工具链和 IDE 支持弱于 C# |
| 学习成本 | ★★★★★ | 语法简洁直观，有 Python 经验的开发者即学即用 |

**本项目适用性：** 本项目的核心计算（伤害公式、生长模拟）为中等复杂度，GDScript 完全胜任。仅在极高频循环（每帧数千个方块的排序更新）可能需要 GDExtension 介入。

#### C#（Godot 环境）

| 维度 | 评分 | 说明 |
|------|------|------|
| 性能 | ★★★★★ | JIT/AOT 编译，数值运算性能优异 |
| 开发效率 | ★★★★☆ | 强类型利于大型项目，但 Godot 4 C# 仍需编译等待（虽已大幅改善） |
| 维护性 | ★★★★★ | 强类型 + IDE 重构支持，长期维护优势明显 |
| 生态 | ★★★★★ | .NET 生态庞大，可复用大量 NuGet 库 |
| 学习成本 | ★★★☆☆ | 对纯 GDScript 开发者有额外学习成本 |

**本项目适用性：** 如果团队有 C# 经验，C# 是强有力的补充选项。但 Godot 4 的 C# 模块需要额外的 .NET SDK 安装和构建步骤，且部分 Godot 4 独有的 GDScript 特性（如 `@export`、信号绑定）在 C# 中支持不够直观。

#### C++ (GDExtension)

| 维度 | 评分 | 说明 |
|------|------|------|
| 性能 | ★★★★★ | 原生编译，极致性能 |
| 开发效率 | ★★☆☆☆ | 编译-链接循环，开发速度远低于 GDScript/C# |
| 维护性 | ★★★☆☆ | 手动内存管理，无 GC，代码量是 GDScript 的 3~5 倍 |
| 生态 | ★★★☆☆ | 仅用于性能关键路径，不适合全栈 |
| 学习成本 | ★☆☆☆☆ | C++ 本身即是高门槛语言 |

**本项目适用性：** 仅推荐作为 **性能关键子系统** 的加速手段（如大规模方块渲染排序、碰撞检测的热路径），不推荐作为主要开发语言。

### 4.3 语言选型策略

推荐 **GDScript 为主，GDExtension 为辅** 的混合策略：

```
项目代码 = 95% GDScript + 5% GDExtension (C++)
         |                 |
         核心逻辑           性能热路径
         （常规开发）         （按需优化）
```

## 5. 后端架构与存储方案

### 5.1 后端需求分析

| 需求类型 | 需求项 | 优先级 |
|----------|--------|--------|
| 必有 | 本地存档（多槽位） | P0 |
| 必有 | 本地设置持久化 | P0 |
| 必有 | Steam 成就 | P1 |
| 可选 | Steam 云存档 | P2 |
| 可选 | 排行榜 | P3 |
| 可选 | 数据统计/分析 | P3 |
| 未来 | 创意工坊支持 | P4 |

### 5.2 本地存储方案

| 方案 | 格式 | 性能 | 可读性 | 版本兼容 | 推荐度 |
|------|------|------|--------|----------|--------|
| **ResourceSaver (二进制)** | `.res` / `.tres` | ★★★★★ | ★★☆☆☆ | ★★★★★ | ★★★★★ |
| JSON | `.json` | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★★☆ |
| ConfigFile | `.cfg` | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| SQLite | `.db` | ★★★★★ | ★★☆☆☆ | ★★★★☆ | ★★☆☆☆ |

**推荐方案：**
- **存档数据：** `ResourceSaver`（二进制 `.res`）- 原生支持 Godot Resource 序列化，版本控制通过 `SaveData.version` 字段实现
- **设置数据：** `ConfigFile`（`.cfg`）- 人类可读，方便调试，Godot 内置 API
- **游戏数据（只读）：** JSON 导入 + 自定义 Resource - 平衡编辑友好与运行时效率

### 5.3 云端方案

| 方案 | 适用场景 | 成本 | 复杂度 | 推荐度 |
|------|----------|------|--------|--------|
| **Steamworks SDK** | 成就/云存档/排行榜 | 免费（需 Steam 上架费） | 中 | ★★★★★ |
| **Godot 内置 HTTP** | 自定义后端通信 | 自建服务器成本 | 高 | ★★★☆☆ |
| **Firebase** | 实时数据/分析 | 按量计费 | 中 | ★★☆☆☆ |
| **PlayFab** | 托管后端 | 按 MAU 计费 | 中 | ★★☆☆☆ |

**推荐方案：** 以 **纯本地单机** 为主，Steamworks SDK 为唯一后端依赖。不引入独立的云后端服务，避免增加复杂度和运维成本。

## 6. 关键模块技术选型细评

### 6.1 2.5D 方块系统

| 方案 | 实施方式 | 引擎支持 | 推荐度 |
|------|----------|----------|--------|
| **自定义 Sprite2D + Y-Sort** | 每个方块用 Sprite2D 节点，手动管理坐标变换和排序 | Godot 原生 Y-Sort | ★★★★★ |
| TileMap isometric | 使用 TileMap 的等距模式 | Unity Tilemap / Godot TileMap 有限支持 | ★★★☆☆ |
| 单个大 Texture 预渲染 | 将整个场景绘制成一张大图 | 任何引擎 | ★☆☆☆☆ |

**结论：** 自定义 Sprite2D + Y-Sort 最适合本项目。原因：
- 方块需要动态放置/破坏，预渲染不灵活
- TileMap 的等距模式不支持本项目的三面纹理 + 高度层系统
- Godot 的 Y-Sort 和 `z_index` 系统完全满足排序需求

### 6.2 符箓绘制输入系统

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| **方向序列识别** | 记录按键/鼠标滑动方向序列，与预设模板匹配 | ★★★★★ |
| 鼠标轨迹匹配 | 识别鼠标绘制轨迹形状 | ★★★☆☆ |
| 组合键释放 | 按住修饰键+技能键释放 | ★★☆☆☆ |

**结论：** 方向序列识别最符合设计文档中"按顺序输入方向键"的方案。Godot 的 `Input` 系统可轻松捕获方向输入，自定义匹配算法实现简单。

### 6.3 伤害计算引擎

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| **GDScript 函数计算** | 直接使用 GDScript 实现多层乘法公式 | ★★★★★ |
| 数据表驱动 | 将公式参数外置到 JSON/Resource 中 | ★★★★☆ |
| C++ GDExtension 加速 | 将高频计算迁移到原生代码 | ★★★☆☆ |

**结论：** GDScript 函数计算即可满足需求。伤害计算虽有多层公式，但调用频率受限于玩家攻击频率（最高每秒几次到十几次），不会成为性能瓶颈。

### 6.4 篆路自动化系统

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| **信号图计算** | 用图数据结构管理灵路连接，信号传播使用拓扑排序更新 | ★★★★★ |
| 事件驱动 | 每个输入变化触发级联更新 | ★★★★☆ |
| 每帧扫描 | 每帧遍历所有节点更新信号 | ★★★☆☆ |

**结论：** 信号图计算 + 事件驱动混合最为合适。灵感来自 Factorio 的电路网络实现，Godot 的 Dictionary 和 Array 足以构建图结构，信号传播算法可在 GDScript 中高效实现。

### 7.2 推荐理由（关键决策点）

#### 引擎选择：Godot 4.x

1. **架构高度匹配：** 设计文档中的 SignalBus / GameManager 三层架构是 Godot 节点树 + 信号系统的原生表达。如果用 Unity，需要额外实现事件总线（如 C# event/UnityEvent），且缺少等同 Godot 节点树的模块化组织方式。

2. **2D 渲染开销最小化：** 320x180 的像素渲染对 Godot 而言是"舒适区"。Godot 的 2D 渲染器是一个独立的轻量管线，没有 Unity 那种为了 3D 准备的 SRP/URP 开销。项目构建体积可控制在 50MB 以内（不含资源）。

3. **Y-Sort + z_index 完美支持 2.5D：** 设计文档中的 `get_sort_key(gx+gy, gz)` 排序逻辑可直接映射到 Godot 的 Y-Sort 和 `z_index` 属性，无需任何插件或 hack。

4. **零成本 + MIT 许可：** 单人独立开发者无需担心 Unity 的许可政策变动（如 2023 年的 Runtime Fee 风波），也无 GameMaker 的月费负担。

5. **场景驱动的开发流程：** Godot 的场景 = 可复用的节点组合，与设计文档中的模块划分（player.tscn、enemies/、items/等）完全一致。

#### 语言选择：GDScript 为主

1. **零摩擦迭代：** GDScript 的热重载能力对于单人开发者的效率提升是显著的，尤其是调试复杂的战斗逻辑和 UI 交互时。

2. **引擎 API 零隔阂：** GDScript 可以直接使用 Godot 的所有 API 特性（信号、`@export`、`@onready`、Group 操作等），而 C# 绑定在某些边缘场景仍有延迟。

3. **性能足够：** 本项目的计算密集型场景（伤害公式、生长模拟、物品处理）都是低频或中等频率，GDScript 完全胜任。唯一的高频场景（方块排序）可通过 GDExtension 按需加速。

### 7.3 备选方案（退路计划）

| 场景 | 备选方案 | 切换成本 |
|------|----------|----------|
| GDScript 性能不足（方块系统） | 用 GDExtension C++ 重写 `BlockRenderer.update_z_indices()` | 低（单个模块替换） |
| 需要更成熟的 UI 系统 | 考虑 Godot 4.3+ 的新 UI 特性或第三方 UI 插件 | 低 |
| Steam 集成困难 | 使用 GDS 的 GodotSteam 或 GDNative 封装 | 中 |
| 需要多人/联机功能 | 考虑 Godot 的 ENet 内置网络或第三方方案 | 高（架构级变更） |

虽然要学新语言，但 GDScript 和 JavaScript 的思维模型非常相似：

| 概念 | JavaScript | GDScript |
|------|-----------|----------|
| 变量 | `let x = 5` | `var x = 5` |
| 常量 | `const MAX = 10` | `const MAX = 10` |
| 函数 | `function foo(bar) { }` | `func foo(bar):` |

**核心差异** 只是语法风格（JS 花括号 vs Python 缩进），编程概念几乎全部通用。
