## 1. 核心结论

### 一个数字：90%

> **自研类引擎的开发量，90% 花在「让不同库协同工作」的胶水代码上，只有 10% 是真正的「游戏逻辑」。**

### 一句话回答

> **作为一个无游戏经验的前端开发者，自研类引擎完成本项目的可行性约为 20%——技术上可以实现，但周期将从原本的 12~18 个月延长到 36~48 个月，且项目可能在中途因架构失控而搁浅。**

### 三个关键事实

| 事实 | 说明 |
|------|------|
| **引擎开发 ≠ 游戏开发** | 你将花 1~2 年造引擎，然后才开始做游戏。而 Cocos Creator 已经把引擎做了 10 年。 |
| **库 ≠ 引擎** | 10 个库拼不出一个引擎。缺少的是：统一生命周期、场景图、序列化、工具链、编辑器。 |
| **单人自研引擎的成功率** | 独立游戏历史上，单人自研引擎最终发布的成功率 < 5%。成功的案例（如《Stardew Valley》）用的也是现成引擎（XNA/MonoGame）。 |

### 2.2 渲染层详解

#### PixiJS（首选候选）

```
优势: WebGL 2D 渲染性能极佳，API 简洁，社区庞大
劣势: 只是渲染器，无场景管理，无生命周期，无碰撞
价格: 免费 (MIT)
```

```javascript
// PixiJS 渲染示例 - 只负责画图，不管别的
import * as PIXI from 'pixi.js';

const app = new PIXI.Application({
    width: 960,
    height: 540,
    backgroundColor: 0x000000
});
document.body.appendChild(app.view);

// 创建方块精灵
const block = PIXI.Sprite.from('block_grass.png');
block.x = 100;
block.y = 200;
app.stage.addChild(block);

// 没有内置的：
// - 每帧 update(deltaTime) 循环 ❌ 需要自己实现
// - 节点树/场景图 ❌ 只有扁平的 stage
// - 坐标变换 ❌ 2.5D 公式要自己写
// - 碰撞检测 ❌ 需要集成 Matter.js
// - 生命周期 ❌ 没有 start/onDestroy
```

#### 关键集成问题

渲染器（PixiJS）和物理引擎（Matter.js）之间的坐标同步：

```javascript
// 每帧你需要手动同步 PixiJS 和 Matter.js 的世界
function syncPhysicsAndRender() {
    // Matter.js 物理体列表
    const bodies = Matter.Composite.allBodies(engine.world);
    
    bodies.forEach(body => {
        // 找到对应的 PixiJS 精灵
        const sprite = body.gameObject; // 自定义引用
        
        // 手动同步位置！
        // 问题：Matter.js 坐标系是 y 向上，PixiJS 是 y 向下
        // 问题：Matter.js 角度是弧度，PixiJS 精灵旋转要转换
        // 问题：物理引擎的更新频率和渲染帧率可能不同
        sprite.x = body.position.x;
        sprite.y = app.screen.height - body.position.y; // 翻转 Y 轴！
        sprite.rotation = body.angle;
    });
}
```

> **这只是一个库的同步问题。当你有 7~8 个库时，这样的胶水代码会指数级增长。**

### 2.3 UI 系统 - 最大的坑

#### 选项 A：HTML/CSS overlay（前端工程师最自然的想法）

```html
<!-- 游戏 Canvas 上覆盖 HTML UI -->
<div id="game-container">
    <canvas id="game-canvas"></canvas>  <!-- PixiJS 渲染到这里 -->
    <div id="ui-overlay">               <!-- 你的 Vue UI -->
        <HUD :hp="player.hp" :mp="player.mp" />
        <InventoryPanel v-if="showInventory" />
        <DialogBox :text="dialogText" v-if="inDialog" />
    </div>
</div>
```

**看似美好，实则噩梦：**

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| **输入竞争** | 🔴 致命 | Canvas 和 HTML 同时响应点击，需要复杂的 hit-test 冒泡处理 |
| **坐标对齐** | 🔴 致命 | Canvas 内部是游戏坐标，HTML 是 CSS 坐标，缩放/全屏时对不齐 |
| **渲染层级** | 🟡 中 | 粒子特效在 Canvas 中，UI 在 HTML 层，中间不能穿插 |
| **性能** | 🟡 中 | HTML reflow 可能引起布局抖动，游戏帧率不稳 |
| **WebGL 覆盖** | 🔴 致命 | 某些 GPU 下 HTML 元素覆盖 Canvas 会导致 WebGL 渲染异常 |
| **手柄导航** | 🔴 致命 | 浏览器 HTML 元素和游戏输入无法统一处理手柄 |

> **实际案例：** 许多用 HTML overlay 做 UI 的游戏项目，最终都重写为 Canvas 内 UI，因为跨层的交互问题太多。

#### 选项 B：Canvas 内 UI（自研）

你需要自研一整套 UI 系统：

```javascript
// 你需要自己实现的 UI 系统最小功能集
class UIButton {
    constructor(x, y, width, height, text) {
        // 1. 碰撞检测（鼠标是否在按钮范围内）
        // 2. 纹理渲染（九宫格背景 + 文字）
        // 3. 状态管理（normal/hover/pressed/disabled）
        // 4. 事件处理（click/hover/拖拽）
        // 5. 动画（按钮弹动效果）
        // 6. 布局（相对定位、自适应缩放）
        // 7. 焦点管理（键盘导航 Tab 顺序）
        // 8. 无障碍（屏幕阅读器支持）
    }
}
```

**工作量估算：** 一个可用的 Canvas 内 UI 系统约 **3000~5000 行代码**，仅基础组件（Button/Label/Panel/ScrollView/Input）。

### 2.4 场景管理 - 另一个巨坑

Cocos Creator 的 `director.loadScene()` 背后做了：

```javascript
// 你需要在自研引擎中实现这些
function loadScene(sceneName) {
    // 1. 卸载当前场景所有节点
    //     - 依次调用每个节点的 onDestroy() 生命周期
    //     - 移除所有事件监听，防止内存泄漏
    //     - 释放已加载的资源引用计数
    //     - 停止所有动画/粒子/Tween
    
    // 2. 清理渲染管线
    //     - 清空 PixiJS 的 stage
    //     - 重置物理引擎世界
    //     - 重置音频上下文
    //     - 清理碰撞检测空间
    
    // 3. 加载新场景资源
    //     - 解析场景描述文件（JSON/XML）
    //     - 递归实例化所有节点和组件
    //     - 加载场景引用的所有纹理/音频/预制体
    //     - 等待异步资源加载完成
    
    // 4. 初始化新场景
    //     - 按正确顺序调用每个节点的 start()
    //     - 启动物理模拟
    //     - 恢复游戏循环
    
    // 5. 场景切换过渡
    //     - 淡入/淡出效果
    //     - 渐进式加载进度条
}
```

**工作量估算：** 约 **2000~4000 行代码**，这是 Cocos Creator 数千工程师多年的成果。

### 3.2 引擎胶水代码总量

```
渲染 (PixiJS)        500行   ← PixiJS 和 Matter.js 坐标同步
物理 (Matter.js)     500行   ← 物理体与游戏对象绑定

输入 (Hammer.js)     300行   ← 输入抽象层

-------------------------------------
胶水代码小计:        ~1800行

核心架构（上表）:     ~13000~23000行
-------------------------------------
自研引擎总代码量:     ~15000~25000行
```

> **对比：** 本项目游戏逻辑本身（按照设计文档的复杂度）估约 **30000~50000 行 GDScript/JavaScript**。这意味着自研引擎的代码量将达到游戏逻辑本身的 **50%~80%**。

### 3.3 编辑器/工具链 - 真正的无底洞

Cocos Creator 提供的不只是运行时，还有整个工具链：

| Cocos Creator 提供的 | 自研方案 |
|---------------------|---------|
| **可视化场景编辑器** | ❌ 需要手写 JSON 场景文件，或者自研编辑器 |
| **拖拽式节点树** | ❌ 代码中硬编码节点层级关系 |
| **Inspector 属性面板** | ❌ 改一个数值需要修改代码 |
| **资源拖拽导入** | ❌ 手动处理资源导入和格式转换 |
| **Prefab 预制体系统** | ❌ 需要自研预制体序列化 |
| **Animation 编辑器** | ❌ 用代码写动画（如 GSAP） |
| **TileMap 编辑器** | ❌ 使用 Tiled 编辑器导出，再自己加载 |
| **粒子编辑器** | ❌ 用代码配置粒子参数 |
| **调试面板** | ❌ 需要自建调试工具 |
| **构建/打包** | ❌ 需要自己配 webpack/electron-builder |
| **热更新** | ❌ 需要自研热更新方案 |

**如果决定自研编辑器，工作量将是运行时引擎的 3~5 倍。**

### 4.2 Electron 方案的关键问题

```javascript

        width: 960,

            // 问题 2: 需要开启 GPU 加速

            contextIsolation: false

    // 问题 5: 游戏全屏时，Electron 菜单栏、快捷键冲突
    // 问题 6: 游戏手柄（Gamepad API）在 Electron 中兼容性差
```

**Electron 游戏的主要问题：**

| 问题 | 严重度 | 说明 |
|------|--------|------|
| **安装包体积** | 🟡 | 最小配置 ~80MB，完整 ~200MB（Cocos Creator 桌面包 ~30MB） |
| **GPU 兼容性** | 🔴 | 某些显卡的 WebGL 实现和 Electron 的 Chromium 有兼容问题 |
| **手柄输入** | 🔴 | Gamepad API 在 Electron 中不稳定，需要 C++ 原生模块 |
| **Steam 集成** | 🔴 | steamworks.js 需要编译原生 .node 文件，构建环境要求高 |
| **帧率稳定性** | 🟡 | Chromium 的 GC 可能导致游戏卡顿 |
| **内存占用** | 🟡 | Electron 基础占用 ~200MB RAM，游戏内容之外的开销 |
| **自动更新** | 🟡 | 需要 electron-updater 配合 |

### 5.2 开发效率对比：实现一个「玩家移动」

#### Cocos Creator (10 行代码)

```javascript
// PlayerController.js - Cocos Creator
import { _decorator, Component, input, Input, EventKeyboard, KeyCode, Vec3 } from 'cc';
const { ccclass, property } = _decorator;

@ccclass("PlayerController")
export class PlayerController extends Component {
    @property speed = 200;

    start() {
        input.on(Input.EventType.KEY_DOWN, this.onKey, this);
    }

    onKey(event) {
        const dir = { w: Vec3.UP, s: Vec3.DOWN, a: Vec3.LEFT, d: Vec3.RIGHT }[event.keyCode];
        if (dir) this.node.translate(dir.multiplyScalar(this.speed * 0.016));
    }
}

// 步骤：
// 1. 创建场景
// 2. 拖入 Player 节点
// 3. 附加 PlayerController 脚本
// 4. 运行
// 用时：~10 分钟
```

#### 自研方案 (需要先搭框架)

```javascript
// 第一步：搭建引擎核心 (15000~25000 行)
// ...

// 第二步：实现玩家移动 (假设引擎已搭建完成)
class PlayerSystem {
    constructor(scene, pixiApp, matterEngine, inputManager) {
        this.scene = scene;
        this.app = pixiApp;
        this.engine = matterEngine;
        this.input = inputManager;
        
        // 创建渲染精灵
        this.sprite = new PIXI.Sprite(texture);
        this.app.stage.addChild(this.sprite);
        
        // 创建物理体
        this.body = Matter.Bodies.rectangle(x, y, 32, 48);
        Matter.Composite.add(this.engine.world, this.body);
        
        // 建立双向引用
        this.body.gameObject = this;
        this.sprite.gameObject = this;
        
        // 注册到场景管理系统
        scene.registerEntity(this);
    }
    
    update(deltaTime) {
        // 处理输入
        if (this.input.isKeyDown('w')) {
            Matter.Body.setVelocity(this.body, { x: 0, y: -speed });
        }
        // ... 其他方向
        
        // 同步物理到渲染
        this.sprite.x = this.body.position.x;
        this.sprite.y = appHeight - this.body.position.y;
    }
    
    onDestroy() {
        this.app.stage.removeChild(this.sprite);
        Matter.Composite.remove(this.engine.world, this.body);
        this.scene.unregisterEntity(this);
    }
}

// 步骤：
// 1. 实现引擎核心框架 (3~6个月)
// 2. 实现 ECS 或对象系统
// 3. 实现场景管理器
// 4. 实现输入管理器
// 5. 实现物理-渲染同步
// 6. 实现生命周期管理
// 7. 编写 PlayerSystem
// 用时：~6 个月（第一次能在屏幕上看到角色移动）
```

## 7. 本项目的 2.5D 方块系统自研复杂度

### 7.2 自研方案的排序实现对比

```javascript
// ===== 自研方案：手动排序 =====
// 需要自己实现一个稳定的排序 + 渲染顺序管理

class DepthSorter {
    constructor() {
        this.blocks = new Map();  // key: "gx,gy,gz" -> Sprite
        this.sortDirty = true;
    }
    
    addBlock(gx, gy, gz, sprite) {
        const key = `${gx},${gy},${gz}`;
        this.blocks.set(key, sprite);
        this.sortDirty = true;
    }
    
    removeBlock(gx, gy, gz) {
        const key = `${gx},${gy},${gz}`;
        this.blocks.delete(key);
        this.sortDirty = true;
    }
    
    updateSortOrder() {
        if (!this.sortDirty) return;
        
        // 设计文档中的排序键
        const sorted = [...this.blocks.entries()].sort((a, b) => {
            const [gx1, gy1, gz1] = a[0].split(',').map(Number);
            const [gx2, gy2, gz2] = b[0].split(',').map(Number);
            return (gx1 + gy1) * 100 + gz1 - (gx2 + gy2) * 100 + gz2;
        });
        
        // 重新排列 PixiJS 容器的子元素顺序
        sorted.forEach(([key, sprite], index) => {
            // PixiJS 的 zOrder 需要手动管理
            sprite.zIndex = index;
        });
        
        // 通知 PixiJS 重新排序
        this.container.sortableChildren = true;
        
        this.sortDirty = false;
    }
}

// ===== Cocos Creator 方案 =====
// 创建方块时设置 Sorting2D 组件即可
const sorting = block.addComponent(Sorting2D);
sorting.sortingOrder = (gx + gy) * 100 + gz;

// 更新时只需重新赋值
function onBlockChanged(gx, gy, gz) {
    // 找到对应的方块节点
    const block = find(`Canvas/World/Block_${gx}_${gy}_${gz}`);
    if (block) {
        block.getComponent(Sorting2D).sortingOrder = (gx + gy) * 100 + gz;
    }
}
```

### 8.2 如果你确实想尝试自研

**建议路径（而非一步到位做《云汲仙田录》）：**

```
学习阶段 (6 个月):
  Step 1: 用 PixiJS 做一个打砖块游戏       → 理解渲染循环
  Step 2: 集成 Matter.js 做物理游戏        → 理解物理同步
  Step 3: 加 Howler.js 做音效             → 理解音频集成
  Step 4: 实现简单的 ECS 框架              → 理解架构
  Step 5: 做一个 10x10 的 2.5D 方块场景    → 理解核心难点

实践阶段 (12 个月):
  Step 6: 用自研框架做一个完整的迷你游戏    → 验证工程可行性

只有完成了以上步骤，才应该考虑：
  Step 7: 用自研框架做《云汲仙田录》       → 36~48 个月
```

### 8.3 最终建议

> **无论从哪个角度看，自研方案对于本项目都是不推荐的。**

| 决策 | 建议 | 理由 |
|------|------|------|
| 🥇 **第一选择** | **Cocos Creator** | JS 零学习成本、中文生态最佳、2D 引擎成熟、开发周期 12~18 个月 |
| 🥈 **第二选择** | **Godot 4.x** | 2.5D 支持最佳、项目架构已设计、开发周期 12~18 个月 |
| ❌ **不建议** | **自研组合方案** | 开发周期 36~48 个月、搁浅概率 60~80%、需同时精通 8+ 库 |

### 8.4 一个形象的比喻

```
用 Cocos Creator 做游戏就像是：
  你买了一辆完整的汽车（引擎、底盘、轮胎都已装好）
  你只需要学会开车（学 Cocos API）
  然后就可以上路了

自研方案做游戏就像是：
  你买来发动机（PixiJS）、变速箱（Matter.js）、轮胎（Howler.js）...
  然后要自己设计底盘（ECS 框架）
  自己焊接车身（UI 系统）
  自己安装电路（场景管理）
  自己调试刹车（生命周期）
  等车能开动的时候，已经过去两年了
  而 Cocos Creator 的玩家已经通关了你的游戏
```

> **除非你的目标是「造一辆车」而非「开到目的地」，否则请使用现成引擎。**
