# 等轴渲染装配单精灵方案（方案 A）

> **对应项目：** 《云汲仙田录》
> **关联模块：** [`BlockSprite.mjs`](../src/engine/render/BlockSprite.mjs) · [`IsoTextureTransformer.mjs`](../src/engine/loader/IsoTextureTransformer.mjs) · [`等轴方块纹理变换系统设计.md`](等轴方块纹理变换系统设计.md)
> **设计决策日期：** 2026-05-09
> **状态：** ✅ 已确认执行

---

## 一、问题背景

当前 2.5D 等轴方块渲染采用**三面独立 Sprite 模式**：每个物块由 3 个 PIXI.Sprite（顶面/左面/右面）组成，通过 `_layoutIsoFaces()` 在运行时计算每个 Sprite 的 anchor/position/scale 来拼合。

该模式存在 ~1px 的接缝间隙问题：顶面菱形边界与左/右平行四边形侧面的边缘无法在像素级紧密贴合。经过多轮迭代修复（纹理边缘填充 `expandTopFaceEdges` + Sprite 级缩放 `TOP_SCALE`），仍未完全消除。

---

## 二、方案 A 核心思路

**将渲染模式从「每个物块 3 个 Sprite」切换为「每个物块 1 个 Sprite」。**

该 1 个 Sprite 使用 `assembleBlock()` 预烘焙的完整等轴方块纹理（顶面 + 左面 + 右面已在像素级装配为一张 ImageData）。Y-sort 排序使高层物块的纹理自然覆盖低层物块，侧边因网格偏移自然可见。

### 本质变化

| 维度 | 三面独立模式（当前） | 装配单精灵模式（方案 A） |
|------|-------------------|----------------------|
| 每物块 Sprite 数 | 3 | 1 |
| 每物块 Draw Calls | 3 | 1 |
| 纹理数量 | 3 张（24×12, 12×21, 12×21） | 1 张（~34×28） |
| 对齐方式 | 运行时浮点位置计算（_layoutIsoFaces） | 像素级预烘焙（assembleBlock） |
| 接缝来源 | 浮点偏移 + 纹理采样偏差 | 预烘焙像素偏移（可一次修复） |
| Y-sort 兼容性 | 按容器排序，3 个子 Sprite 一同排序 | 按 Sprite 排序，同现有机制 |
| 纹理变换管线 | 完全相同（rotate → compress → shear） | 完全相同，额外一步 `assembleBlock()` |

### 关键发现

**两种模式已在代码中并行存在**，通过 `createWithIsoTransform({ useAssembled: true/false })` 切换。方案 A 不是重新实现，而是：

1. 修复 `assembleBlock()` 中的像素级对齐精度
2. 将 `useAssembled` 默认值从 `false` 翻转为 `true`
3. 保留三面模式作为降级/调试选项

## 三、纹理管线对比图

```
三面独立模式：
  源纹理(16×16) → rotateTexture45 → compressVerticalDiagonal → expandTopFaceEdges → top(24×12)
  源纹理(16×16) → shearToParallelogram(left)  → left(12×21)
  源纹理(16×16) → shearToParallelogram(right) → right(12×21)
                               ↓
                   BlockSprite._layoutIsoFaces()
                   (3 个 Sprite, 运行时对齐)

装配单精灵模式：
  源纹理(16×16) → rotateTexture45 → compressVerticalDiagonal → expandTopFaceEdges → top(24×12)
  源纹理(16×16) → shearToParallelogram(left)  → left(12×21)
  源纹理(16×16) → shearToParallelogram(right) → right(12×21)
                               ↓
                   assembleBlock(top, left, right) ← 新增/修复对齐
                               ↓
                   BlockSprite.setAssembledTexture()
                   (1 个 Sprite, 像素级预对齐)
```

纹理变换管线完全相同，**仅渲染阶段的 Sprite 数量和对齐策略不同**。

## 四、预计收益

| 指标 | 三面模式 | 装配模式 | 改善幅度 |
|------|---------|---------|---------|
| Draw Calls（5000 物块） | 15000 | 5000 | **-66%** |
| 纹理像素总量 | 24×12 + 12×21 + 12×21 = 828px | ~34×28 = 952px | +15%（含空白边距，可优化） |
| 纹理上传次数（加载时） | 3 次 / 物块类型 | 1 次 / 物块类型 | -66% |
| Sprite 管理复杂度 | 高（3 个 Sprite 的 anchor/position/scale 协调） | 低（1 个 Sprite, anchor(0.5,0)） | 显著降低 |
| 接缝消除机制 | 运行时浮点参数调优 | 像素级数据确定 | **彻底消除** |

## 五、实施步骤

### 5.1 修复 `assembleBlock()` 对齐精度

检查并修正 `assembleBlock()` 中顶面与侧面的像素级偏移计算：

```
当前核心代码（IsoTextureTransformer.mjs:386-432）:
- topOffsetX/Y: 顶面在装配画布中的位置
- leftOffsetX/Y: 左面位置（基于 topCenterX 计算）
- rightOffsetX/Y: 右面位置（基于 topCenterX 计算）
- sideShiftY: Math.ceil(SHEAR_OFFSET) = 6（必须与 _layoutIsoFaces 一致）
```

可能的偏差来源：
- `computeBoundingBox` 返回的 `minX/minY/width/height` 在四舍五入时的 1px 偏差
- `leftOffsetX = topOffsetX + topCenterX - leftBB.minX - leftBB.width` 的水平对齐偏差
- `leftOffsetY = topBB.height - leftBB.minY - sideShiftY` 的垂直对齐偏差

修复策略：通过可视化测试（将装配结果绘制到 Canvas）确认每个物块类型的对齐质量。

### 5.2 翻转 useAssembled 默认值

```javascript
// BlockSprite.mjs:744-745
static async createWithIsoTransform(options = {}) {
    const { useAssembled = true, ...rest } = options;  // false → true
```

### 5.3 添加可选的顶面微扩

在 `setAssembledTexture()` 中，保留对装配后纹理的微扩能力（类似当前 `_layoutIsoFaces` 的 `TOP_SCALE`），作为兜底保险：

```javascript
// 在 setAssembledTexture 中增加 scale 参数
setAssembledTexture(assembledData, { scaleX = 1, scaleY = 1 } = {}) {
    // ...
    this._assembledSprite.scale.set(scaleX, scaleY);
}
```

### 5.4 更新测试

- 修复 `BlockRenderer.test.mjs` 中与装配模式相关的 mock 尺寸
- 在 `IsoTextureTransformer.test.mjs` 中增加 `assembleBlock()` 对齐精度测试
- 新增端到端测试：验证装配模式下的 Y-sort 视觉正确性

### 5.5 保留三面模式为调试选项

通过 `createWithIsoTransform({ useAssembled: false })` 或 `block.showSeparateFaces()` 可在需要时切换回三面模式，用于调试和对比。

## 六、风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 预烘焙纹理无法在运行时更换单面贴图 | 物块类型级预烘焙，每种 blockType 在加载时烘焙一次 |
| 装配纹理尺寸略大于单面，影响纹理集打包 | 优化 `_cropToContent` 减少边距，或使用紧凑纹理图集 |
| 若对齐偏差仍存在，修复需重新烘焙所有物块类型 | 确保 `assembleBlock()` 对齐函数为纯数据变换，修复后自动生效 |

## 七、与现有架构的一致性核查

- ✅ **ES Modules** — 全部使用 import/export
- ✅ **零依赖核心** — 不引入新第三方库
- ✅ **模块边界** — 纹理变换在 `loader/`，渲染在 `render/`，职责分离
- ✅ **不污染 window** — 全部通过模块导出
- ✅ **EventBus 解耦** — Sprite 创建不直接依赖渲染系统
- ✅ **每个文件 < 300 行** — 现有文件拆分合理
